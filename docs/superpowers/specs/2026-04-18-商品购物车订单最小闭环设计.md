# 商品、购物车与订单最小闭环设计

## 1. 背景

当前项目已经具备单体模块化骨架，但还没有第一个真正可用的业务闭环。为了尽快进入可演示、可迭代的 MVP 阶段，需要优先打通最小交易链路。

本次目标明确收敛为：

- 商品使用无 SKU 的简单模型
- 购物车以 Redis 作为主存
- 创建订单时将购物车勾选项快照落到 MySQL
- 不做支付、不做库存、不做优惠、不做复杂状态流转

## 2. 本次目标

交付一个最小可用下单闭环：

1. 查询商品列表
2. 查看商品详情
3. 将商品加入购物车
4. 查看购物车
5. 修改购物车商品数量
6. 删除购物车商品
7. 勾选购物车项并创建订单

订单创建成功后：

- 订单状态固定为 `CREATED`
- 订单与订单明细落到 MySQL
- Redis 中已下单的购物车项被移除

## 3. 非目标

本次不实现以下内容：

- 支付能力
- 库存校验与扣减
- 优惠券、促销价、满减
- SKU、多规格、组合商品
- 分布式事务
- 异步订单处理

## 4. 业务边界

### 4.1 商品模块 `product`

本次商品模型保持极简，只保留下单必需字段：

- `id`
- `name`
- `price`
- `status`

对外能力：

- 查询商品列表
- 查询商品详情
- 批量查询商品快照信息，供订单模块使用

说明：

- 本次不引入 SKU
- 一个商品只有一个价格
- 仅允许状态为 `ON_SHELF` 的商品进入购物车和下单

### 4.2 购物车模块 `cart`

购物车使用 Redis 作为主存，按用户维度存储。

每个购物车项至少包含：

- `productId`
- `productNameSnapshot`
- `priceSnapshot`
- `quantity`
- `checked`

对外能力：

- 加入购物车
- 查看购物车
- 修改商品数量
- 删除购物车项
- 勾选或取消勾选购物车项
- 获取当前用户已勾选的购物车项，供订单创建使用
- 创建订单后移除已下单项

说明：

- 购物车中的商品名和价格保存快照，减少下单瞬间的额外查询复杂度
- 订单创建时仍允许再次校验商品是否有效，但以购物车快照作为订单写入来源

### 4.3 订单模块 `order`

订单模块负责从购物车勾选项生成订单。

本次订单结构至少包含：

- `orders`
  - `id`
  - `user_id`
  - `order_status`
  - `total_amount`
  - `created_at`
  - `updated_at`
- `order_item`
  - `id`
  - `order_id`
  - `product_id`
  - `product_name`
  - `sale_price`
  - `quantity`
  - `line_amount`

订单创建规则：

- 只允许基于“已勾选”的购物车项创建订单
- 如果没有勾选项，拒绝创建
- 创建成功后订单状态固定为 `CREATED`
- 创建成功后从购物车中删除本次已下单项

## 5. 数据存储设计

### 5.1 MySQL

MySQL 存储真实业务数据：

- 商品表 `product`
- 订单表 `orders`
- 订单明细表 `order_item`

本次可根据需要补充或调整表结构，但核心目标不变：商品与订单持久化必须真实落库。

### 5.2 Redis

Redis 存储购物车。

建议键结构：

- `cart:user:{userId}`

值结构建议使用 Hash 或 Redis JSON 风格的序列化对象列表；第一版可以优先选实现最直接的一种，不为未来做过度抽象。

## 6. 接口范围

建议本次最小 API 范围如下：

- `GET /api/v1/products`
- `GET /api/v1/products/{id}`
- `POST /api/v1/cart/items`
- `PUT /api/v1/cart/items/{productId}`
- `DELETE /api/v1/cart/items/{productId}`
- `PUT /api/v1/cart/items/{productId}/checked`
- `GET /api/v1/cart`
- `POST /api/v1/orders`

## 7. 关键流程

### 7.1 商品浏览

1. 用户请求商品列表
2. `product` 模块从 MySQL 读取上架商品
3. 返回商品列表

### 7.2 加入购物车

1. 用户选择商品并传入数量
2. `cart` 模块调用 `product/api` 校验商品是否存在且可售
3. 将商品快照写入 Redis 购物车
4. 返回最新购物车内容

### 7.3 创建订单

1. 用户发起创建订单请求
2. `order` 模块调用 `cart/api` 获取当前已勾选项
3. 校验勾选项非空
4. 根据购物车快照生成 `orders` 与 `order_item`
5. 持久化到 MySQL
6. 调用 `cart/api` 删除本次已下单项
7. 返回订单结果

## 8. 错误处理

本次至少处理以下错误场景：

- 商品不存在
- 商品已下架
- 加入购物车数量非法
- 购物车为空
- 未勾选任何商品却创建订单
- 创建订单时购物车项失效

统一要求：

- 使用统一响应结构
- 使用统一业务异常编码
- 不在 Controller 中写业务规则

## 9. 测试范围

本次应至少覆盖：

- 商品列表查询
- 加入购物车
- 修改购物车数量
- 删除购物车项
- 勾选购物车项
- 基于勾选项创建订单
- 创建订单后购物车项被清理
- 未勾选项创建订单失败

## 10. 演进路径

这次闭环完成后，后续可以自然扩展：

1. 在 `order` 模块中加入模拟支付
2. 在 `product` 模块中引入库存字段与库存校验
3. 将购物车 Redis 结构升级为更高并发友好的模型
4. 为订单创建补充领域事件，驱动 analytics/recommendation
5. 再逐步引入 SKU、优惠、支付记录等能力

## 11. 实施原则

- 优先保证闭环可运行，而不是追求一次做全
- 优先保证模块边界正确，而不是追求一次抽象完整
- Redis 购物车主存策略不变
- 订单必须真实落 MySQL
- 所有新增文档统一使用中文
