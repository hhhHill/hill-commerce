# Feature Specification: admin-product-management

**Feature**: `admin-product-management`  
**Status**: active  
**Parent Baseline**: `specs/hill-commerce-mvp/spec.md`

## Purpose

定义后台商品管理闭环，包括分类管理、商品列表筛选、商品新建/编辑、SPU/SKU 规则、字段语义和后台可用性约束。

本 feature 不仅定义“能否保存商品数据”，还定义后台 `SALES` / `ADMIN` 用户能否在理解字段含义的前提下稳定完成商品维护。

## Scope

### In Scope

- 一级分类管理
- 商品列表筛选与状态操作
- 商品新建与编辑
- 商品 `SPU + SKU` 管理
- 销售属性与展示属性
- 商品图片 URL
- 商品描述
- 上架、下架、逻辑删除
- 后台关键字段的静态问号帮助提示

### Out of Scope

- 多级分类
- 图片上传系统
- 商品审核流
- 批量导入导出
- 高级商品运营能力
- 动态 tooltip 文案

## Roles And Goals

### Sales

- 维护一级分类
- 通过筛选快速定位商品
- 创建和编辑商品
- 管理商品状态、SKU 与展示内容

### Admin

- 拥有与 `Sales` 等价的后台商品管理访问能力
- 以更高权限身份审查与维护后台商品数据

### Customer

- 不可访问后台商品管理能力

## Business Rules

- 只支持一级分类
- 商品采用 `SPU + SKU` 模型
- 每个商品最多支持 2 个销售属性
- 分类状态枚举固定为 `ENABLED`、`DISABLED`
- 商品状态枚举固定为 `DRAFT`、`ON_SHELF`、`OFF_SHELF`
- SKU 状态枚举固定为 `ENABLED`、`DISABLED`
- 商品描述保存为富文本源字符串或 HTML 源字符串
- 图片先采用 URL 录入
- SKU 组合键必须唯一
- SKU 编码允许手工填写；为空时由后端自动生成
- 商品删除采用逻辑删除
- 后台商品管理采用一次提交的聚合保存模型
- 关键输入字段和关键筛选字段必须在标签右侧提供统一的问号帮助提示
- 帮助提示使用静态文案，不根据当前页面状态动态变化
- 帮助提示文案采用两句式：先解释业务含义，再解释规则、限制或默认行为
- 帮助提示用于降低理解成本，不替代后端校验和错误提示

## Page Rules

### Category Management Page

#### Page Goal

让 `Sales` / `Admin` 维护一级分类，不承担商品录入职责。

#### Fields

- `分类名称`
  - 表示后台识别和前台展示使用的一级分类名称
  - 应使用用户可理解的业务名称，而不是内部缩写
- `排序`
  - 表示分类的展示顺序
  - 数值越小越靠前
- `状态`
  - 表示该分类当前是否允许继续被新商品选用
  - 停用不等于删除，历史商品仍可保留分类归属
  - 字面量枚举固定为 `ENABLED` / `DISABLED`

#### Operations

- 可创建分类
- 可编辑分类名称、排序、状态
- 可停用分类
- 可删除未关联任何商品的分类
- 分类一旦关联过商品，即不允许删除，只允许停用

#### Tooltip Coverage

必须为以下字段提供问号提示：

- `分类名称`
- `排序`
- `状态`

#### Feedback Rules

- 创建、编辑、停用成功后，应有明确成功反馈
- 删除失败时，应明确说明是“分类已关联商品，不能删除”
- 校验失败时，应明确指出是名称、排序还是状态问题

### Product List Page

#### Page Goal

作为后台商品查找、筛选、状态判断和进入编辑页的入口，不承担复杂录入任务。

#### Filter Rules

- `分类筛选`
  - 用于按商品所属一级分类过滤结果
- `状态筛选`
  - 用于按商品当前上架状态过滤结果
  - 字面量枚举固定为 `DRAFT` / `ON_SHELF` / `OFF_SHELF`
- `关键词筛选`
  - 用于按商品名称与 `SPU` 编码定位商品
  - 对商品名称采用模糊匹配
  - 对 `SPU` 编码采用精确匹配
  - 同一个关键词对商品名称与 `SPU` 编码采用 `OR` 关系，只要命中任一项即可返回结果
  - 当前交互模式为显式提交筛选，不要求输入时实时搜索，也不要求 debounce

#### List Field Rules

- `分类名称`
  - 显示商品当前所属一级分类
- `商品名称`
  - 显示后台录入的商品主名称
- `状态`
  - 显示当前商品是否上架
  - 字面量枚举固定为 `DRAFT` / `ON_SHELF` / `OFF_SHELF`
- `SPU 编码`
  - 显示商品级识别编码
- `起售价`
  - 显示该商品所有可见 SKU 中的最低售价

#### Operations

- 可进入编辑页
- 可进行上架/下架切换
- 可执行逻辑删除
- 列表页不承担 SKU、属性和描述的复杂编辑

#### Tooltip Coverage

必须为以下关键筛选字段提供问号提示：

- `分类`
- `状态`
- `关键词`

#### Feedback Rules

- 上架/下架成功后应有明确反馈
- 删除成功后默认不再出现在正常列表中
- 删除失败时应明确返回失败原因

### Product Create/Edit Page

#### Page Goal

让 `Sales` / `Admin` 在单页内完成商品基础信息、图片、描述、属性和 SKU 的聚合录入，并在提交前理解各字段含义。

#### Section: Basic Information

包含：

- `分类`
- `商品名称`
- `副标题`
- `商品状态`
- `SPU 编码`

字段规则：

- `分类`
  - 选择商品所属一级分类
- `商品名称`
  - 作为后台商品主名称和前台核心展示名称
- `副标题`
  - 用于补充商品卖点或简要描述
- `商品状态`
  - 定义当前商品是上架还是下架
  - 字面量枚举固定为 `DRAFT` / `ON_SHELF` / `OFF_SHELF`
  - `DRAFT` 表示草稿，`ON_SHELF` 表示已上架，`OFF_SHELF` 表示已下架
- `SPU 编码`
  - 为商品级识别编码，不是面向消费者的显示名称
  - 留空时由后端自动生成

#### Section: Images And Description

包含：

- `封面图 URL`
- `详情图 URL`
- `商品描述`

字段规则：

- `封面图 URL`
  - 用于商品列表和详情主视觉
- `详情图 URL`
  - 用于详情页内容展示
- `商品描述`
  - 保存富文本源字符串或 HTML 源字符串

#### Section: Display Attributes

字段规则：

- 展示属性用于补充商品信息展示
- 展示属性不会生成 SKU 组合
- 例如材质、产地、适用场景等属于展示属性范畴

#### Section: Sales Attributes And SKU

字段规则：

- 销售属性用于定义 SKU 差异维度
- 每个商品最多支持 2 个销售属性
- 销售属性值组合会映射为 SKU 草稿表
- 无销售属性时，也必须允许保存一个默认 SKU
- `SKU 编码` 可手工填写，留空时由后端补码
- `SKU 状态` 仅表示该具体 SKU 是否可用，使用 `ENABLED` / `DISABLED` 语义
- `SKU 状态` 不得复用商品状态枚举，不得使用 `DRAFT` / `ON_SHELF` / `OFF_SHELF`
- `SKU` 组合必须唯一
- `SKU` 的价格、库存、状态属于 SKU 自身属性，不属于 SPU 公共字段
- `SKU` 列表必须提供明确的表头行，至少包括 `组合`、`SKU 编码`、`SKU 价格`、`SKU 库存`、`低库存阈值`、`SKU 状态`
- `SKU` 表头、问号提示和输入控件必须共享同一套列布局，避免字段名与输入框错位

#### Tooltip Coverage

商品新建/编辑页的关键输入字段必须提供问号提示，至少包括：

- `分类`
- `商品名称`
- `副标题`
- `商品状态`
- `SPU 编码`
- `封面图 URL`
- `详情图 URL`
- `商品描述`
- `展示属性名`
- `展示属性值`
- `销售属性名`
- `销售属性值`
- `SKU 编码`
- `SKU 价格`
- `SKU 库存`
- `SKU 状态`

#### Feedback Rules

- 保存成功后应明确提示创建成功或更新成功
- 校验失败时，应指向具体分区和字段
- `SKU` 组合冲突时，应明确提示是组合不唯一
- `SPU` 编码冲突时，应明确提示是商品级编码冲突
- 页面应让用户知道保存是一次聚合提交，而不是分区分别保存

## Cross-Page Rules

- 后台页面默认只允许 `SALES` 与 `ADMIN`
- 术语必须在页面间保持一致，不得同一处写“销售属性”、另一处写“规格项”而未说明等价关系
- 分类、商品、SKU 的状态文案应与实际后端枚举语义一致
- 商品状态与 `SKU` 状态必须严格区分，前者是商品是否上架，后者是具体 SKU 是否可用
- 分类状态、商品状态、SKU 状态都必须在 spec 中保留明确字面量定义，不得只保留“启用/停用”或“上架/下架”的自然语言描述
- 静态问号提示应使用统一样式和交互，不得部分字段使用、部分字段缺失
- 提示文案必须与系统真实行为一致，尤其是自动补码、唯一性限制、停用与删除边界

## Tooltip Copy Policy

- 所有关键字段的标签右侧统一放置问号图标
- 鼠标悬浮时显示静态帮助文案
- 帮助文案格式统一为：
  - 第一句：说明该字段的业务含义
  - 第二句：说明关键规则、限制或默认行为
- 不做状态联动文案
- 不把错误提示、接口报错或技术术语直接当成 tooltip 文案

## Acceptance Criteria

- `Sales` 可创建、编辑、停用一级分类，并理解分类字段含义
- 分类一旦关联过商品，不允许删除，只允许停用
- `Sales` 可通过商品列表页筛选并快速定位商品
- 商品列表中的分类、状态、`SPU` 编码和起售价含义清晰且可验证
- 关键词筛选的匹配语义清晰且可验证：名称模糊匹配、`SPU` 精确匹配、两者为 `OR` 关系
- `Sales` 可在商品编辑/新建页完成分类、基础信息、图片、描述、属性和 SKU 的聚合保存
- 用户能理解 `SPU`、展示属性、销售属性、`SKU` 之间的区别
- 关键输入字段和关键筛选字段旁均存在统一的问号提示
- 问号提示文案能解释字段业务含义，并说明关键规则或限制
- Customer 不可访问后台商品管理能力
