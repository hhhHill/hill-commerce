# Feature Specification: order-checkout

**Feature**: `order-checkout`  
**Status**: active  

## Purpose

定义从购物准备进入真实订单创建的核心交易闭环，覆盖最终结算确认、下单前二次校验、订单创建、订单快照、库存扣减、未支付订单手动取消与库存回补，为后续支付能力提供稳定订单事实源。

## Scope

### In Scope

- 基于已勾选购物车项进入最终结算确认页
- 下单前服务端二次校验
- 订单创建
- 订单主表与订单项落库
- 地址快照
- 商品快照
- 下单即扣减库存
- 未支付订单手动取消
- 取消后的库存回补
- 订单状态历史记录
- 订单提交结果页与最小订单详情查询

### Out of Scope

- 匿名下单
- 商品详情页立即购买
- 真实支付或模拟支付
- 优惠券、促销、复杂运费和复杂价格计算
- 退款、售后
- 订单拆分
- 自动超时关闭未支付订单
- 库存预占高级策略

## User Journeys

### Journey 1: 进入最终结算确认页

登录用户从 `cart-preparation` 的 `/checkout-summary` 进入独立 `/checkout` 页面，查看当前已勾选购物车项、默认地址和最终确认信息。

### Journey 2: 提交订单

登录用户在 `/checkout` 页面提交订单。服务端基于当前用户已勾选购物车项再次校验商品状态、SKU 可售性、库存和默认地址。全部通过后创建订单、订单项、快照和状态历史，并立即扣减库存。

### Journey 3: 查看订单提交结果

订单创建成功后，用户进入订单提交结果页，查看订单号、订单金额、订单状态和去支付入口。

### Journey 4: 取消未支付订单

登录用户可在订单结果页或最小订单详情页取消 `PENDING_PAYMENT` 订单。取消成功后系统回补库存并记录状态历史。

## Checkout Rules

- 下单来源只支持当前用户已勾选购物车项
- 不支持空购物车、未勾选条目或商品详情页立即购买直达下单
- `cart-preparation` 的 `/checkout-summary` 只负责准备态校验与汇总，不负责真正创建订单
- `order-checkout` 新增独立 `/checkout` 页面，负责最终确认与提交订单
- `/checkout` 页面只展示当前已勾选且准备进入下单的购物车项
- `/checkout` 页面应展示商品名称、SKU 规格、单价、数量、小计、合计、默认地址和异常提示
- `/checkout` 不承担地址管理和重新勾选购物车项职责
- 若当前已勾选项为空、全部失效或默认地址缺失，`/checkout` 必须明确提示并阻止提交订单

## Order Creation Rules

- 提交订单只能由登录用户发起
- 服务端只基于当前用户已勾选购物车项创建订单，不允许客户端直接上传任意商品明细作为订单事实源
- 提交时服务端必须重新拉取商品、SKU、库存、默认地址和购物车勾选状态做最终二次校验
- 二次校验至少包括：商品是否可售、SKU 是否有效、SKU 是否启用、库存是否满足、默认地址是否存在且属于当前用户、当前是否确实存在已勾选条目
- 任一条目校验失败，则整笔订单创建失败；首版不支持部分条目成功下单
- 订单创建成功后，参与本次下单的已勾选购物车项应从购物车中移除，避免重复下单
- 若订单创建失败，购物车原有勾选项和数量不应被破坏

## Snapshot Rules

订单创建成功后，系统至少应保存以下快照信息：

- 订单主表：订单号、用户 ID、订单状态、订单总金额、地址快照、提交时间
- 订单项表：商品 ID、SKU ID、商品名称快照、SKU 规格快照、成交单价快照、数量、小计
- 状态历史表：记录订单创建和后续关键状态变化

快照规则：

- 商品快照至少包含商品名称、SKU 规格文本和成交单价
- 地址快照至少包含收货人、联系电话、所在地区和详细地址
- 快照一旦写入订单，不随用户后续修改商品信息或地址而回写变化

## Inventory Rules

- 订单创建成功即扣减库存
- 库存扣减与订单创建必须保持同一事务语义
- 若库存扣减失败，则订单创建必须整体失败
- 取消 `PENDING_PAYMENT` 订单时，系统必须按订单项数量回补对应 SKU 库存
- 库存回补与订单取消必须保持同一事务语义

## Cancellation Rules

- 只有 `PENDING_PAYMENT` 状态订单允许手动取消
- 取消动作只能由订单所属用户发起
- 用户不能取消他人订单
- 订单取消成功后，状态必须更新为 `CANCELLED`
- 已进入 `PAID` 或其他后续状态的订单，不允许再走未支付手动取消规则
- 同一笔订单重复取消时，系统必须具备幂等保护，不能重复回补库存

## Order Status Rules

本期最小订单状态集合为：

- `PENDING_PAYMENT`
- `CANCELLED`
- `PAID`

本 feature 仅定义以下状态流转：

- 创建成功后进入 `PENDING_PAYMENT`
- 手动取消成功后从 `PENDING_PAYMENT` 进入 `CANCELLED`

`PAID` 作为后续 `payment` feature 的目标状态保留，本 feature 不负责把订单推进到 `PAID`。

## API And Page Boundaries

后端最小接口面：

- `GET /api/orders/checkout`
  - 返回当前用户可进入最终确认页的订单准备数据
- `POST /api/orders`
  - 基于当前勾选购物车项创建订单
- `GET /api/orders/{orderId}`
  - 返回订单结果页和最小订单详情所需信息
- `POST /api/orders/{orderId}/cancel`
  - 取消未支付订单

前端最小页面面：

- `/checkout`
  - 最终结算确认页
- `/orders/[orderId]/result`
  - 订单提交结果页
- `/orders/[orderId]`
  - 最小订单详情页

## Business Rules

- 订单创建只消费 `cart-preparation` 中的已勾选购物车项
- 最终有效性以后端二次校验为准，不能信任前端汇总页状态
- 本期不支持部分商品下单成功
- 订单创建与库存扣减必须一致
- 订单取消与库存回补必须一致
- 订单状态变化必须可追踪
- `order-checkout` 在“订单创建成功并等待支付”处结束，支付状态推进由 `payment` 定义

## Acceptance Criteria

- 登录用户可从已勾选购物车项进入独立 `/checkout` 页面
- `/checkout` 页面可展示已勾选条目、默认地址和最终确认信息
- 提交订单时系统会再次校验商品状态、SKU 有效性、库存和默认地址
- 订单创建成功后生成 `PENDING_PAYMENT` 订单，并保存商品快照和地址快照
- 订单创建成功后立即扣减库存
- 订单创建成功后可进入订单提交结果页查看订单号、金额和状态
- 用户可取消 `PENDING_PAYMENT` 订单，取消后系统回补库存
- 非 `PENDING_PAYMENT` 订单不会被视为可取消
- 关键状态变化可在状态历史中追踪

## Boundaries and Dependencies

- 本 feature 依赖 `cart-preparation` 提供已勾选购物车项、默认地址和准备态汇总规则
- 商品可见性、商品详情和 SKU 展示规则由 `product-discovery` 定义
- 登录态与权限事实源由 `auth-permission` 定义
- 支付流水、支付成功推进订单状态、超时关闭等规则由 `payment` 定义
