# Review Notes: admin-product-management

**Review Date**: 2026-05-12
**Reviewed Files**: `spec.md`, `plan.md`, `tasks.md`
**Reviewer**: Claude (AI review, with codebase cross-reference)

---

## 总体评价

三份文档（spec / plan / tasks）结构完整，SPU + SKU 模型定义清晰，页面职责边界明确，Tooltip 规范精细化程度高。但存在几个实质性的遗漏和与当前实现的脱节问题。

---

## P0 - 必须修复

### 1. 分页完全缺失

商品列表页定义了三项筛选项和五项展示字段，但完全没有涉及分页。当前 `ProductAdminService.listProducts()` 的实现是一次性返回全部结果，spec 既没有把分页纳入 scope，也没有放进 out of scope。

**建议：** 明确分页是否在本 feature 范围内。如果是，需定义每页条数默认值、分页参数名和分页交互方式。

### 2. 枚举值未在 spec 中明确定义

spec 反复提到"状态"但从未给出枚举字面量：

| spec 中的表述     | 应有定义                                                          |
| ----------------- | ----------------------------------------------------------------- |
| 分类"启用/停用"   | 未定义枚举值                                                      |
| 商品"上架/下架"   | plan.md 提到 `DRAFT/ON_SHELF/OFF_SHELF` 但 spec.md 未确认         |
| SKU 状态          | spec.md 定义了 `ENABLED/DISABLED`，这一点无问题                   |

plan.md 中提到"商品状态与 SKU 状态不得混用"，但实际上代码中的 `ProductAdminController.updateProductStatus()` 使用 `DRAFT` / `ON_SHELF` / `OFF_SHELF` 作为商品状态，spec.md 中没有对应的枚举定义。这会导致验收时口径不一致。

---

## P1 - 应尽快修复

### 3. 关键词搜索的匹配语义不明确

spec 写"按商品名称与 SPU 编码定位商品"，但未定义：
- 是精确匹配还是模糊搜索？
- 名称和 SPU 编码是 AND 还是 OR 关系？
- 搜索是否需要 debounce？

**建议：** 明确上述三点。

### 4. 商品列表缺少默认排序规则

筛选、展示字段都定义了，但没有定义默认排序（按创建时间？按 SPU 编码？按分类？），这在验收时会成为分歧点。

### 5. 商品描述的编辑器选型未定义

spec 说"保存富文本源字符串或 HTML 源字符串"，但前端实际只有一个 `<textarea>`。这个问题有两种解读：
- 如果只保存 raw HTML 就够，spec 应写明"允许纯文本输入，保存为 HTML"
- 如果需要所见即所得编辑器，spec 应定义选型约束

当前这个模糊点会导致前后端对接时理解不一致。

### 6. 展示属性的结构定义过弱

spec 只写了"例如材质、产地、适用场景等属于展示属性范畴"，但没有定义数据结构。当前实现中 `product_attribute_values` 是自由 key-value，这没问题，但 spec 应确认这一点——否则实现者可能将其理解为预定义属性集。

### 7. SKU 默认行为只有一个句子

"无销售属性时，也必须允许保存一个默认 SKU"——但默认 SKU 的价格、库存、SKU 编码应该如何？这些需要明确定义。

### 8. 逻辑删除的恢复/查看机制缺失

spec 说"删除成功后默认不再出现在正常列表中"，但没有定义：
- 已删除商品能否通过筛选恢复可见？
- 已删除商品能否恢复（undelete）？
- 是否有"回收站"概念？

## P2 - 改进建议

### 9. plan.md 与 spec.md 有大量重复

plan.md 的 "Technical Boundaries" 和 spec.md 的 "Business Rules" 存在多处重叠（SKU 自动补码、独立枚举、聚合保存等），这些不应该出现在两份文档中。

- **plan.md** 应该侧重实现策略、技术选型、交付路径
- **spec.md** 应该侧重业务规则、页面行为、验收标准

### 10. tasks.md 的任务粒度不一致

共 17 项，部分精细（"固化分类删除/停用边界"），部分过于笼统（"定义实现后的回归测试与检查项"没有可操作的子项）。且所有 tasks 都是 unchecked 状态，与代码库已有大量实现的事实不匹配——很多功能已经 build 好了，tasks 没有反映实际进度。

### 11. 缺少并发编辑策略

两个 Sales 同时编辑同一个商品时会发生什么？后提交者覆盖前提交者？还是需要乐观锁？spec 没有涉及。

### 12. 缺少资源数量上限

产品图片数量、展示属性数量、销售属性数量（只定义了 ≤2）、每个商品的 SKU 数量都没有上限约束。

### 13. plan.md 的 Risks 章节缺少缓解措施

列出了 9 个风险但没有任何 mitigations。例如 "SKU 组合生成与已有数据同步复杂度"——打算怎么应对？需要补充对应策略。

### 14. 验收标准的可测试性

"用户能理解 SPU、展示属性、销售属性、SKU 之间的区别"——这是一个主观标准，无法自动化验证。需要转换为可观测的行为指标：

- 用户在分类页/编辑页看到每个关键字段旁都有问号提示
- tooltip 文案符合两句式规范（业务含义 + 规则/限制）
- tooltip 文案与系统真实行为一致

## 已确认正确的部分

以下方面 spec 处理得很好，无需修改：

- SPU + SKU 模型边界清晰，销售属性 / 展示属性 / SKU 三者区分到位
- 分类"关联商品后不可删除，只能停用"的规则完整明确
- Tooltip 规范统一——问号图标位置、两句式格式、静态文案要求——都写得很具体
- Page Rules 的三大页面分别定义 Fields / Operations / Tooltip Coverage / Feedback Rules，结构一致、可操作
- Cross-Page Rules 覆盖了术语一致性、枚举一致性和权限范围
- Out of Scope 明确排除了多级分类、审核流、批量操作等，边界清晰
